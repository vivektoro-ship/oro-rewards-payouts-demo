The Incentive Engine · Funnel Brain Layer 6 · ENG-1680

Attribution earns the money

The funnel already records who earned every loan. This engine reads that record and turns it into money: an append-only accrual ledger, a rulebook the business owns, and a number every actor can watch move in real time — never recomputed by hand.

Every incentive is one sentence: for this kind of actor on this kind of loan, pay this rate on this basis, adjusted by these gates, released on this schedule.

The three accrual states · D11

Firm

Per-loan, locked at disbursal. The actor sees it the instant the loan books — nothing about the period can change it.

Provisional

Period-grain, recomputed as the period fills. Always shown with slab context — you are here, the next slab needs N more.

Final

Frozen at period close. Revisable only by an explicit append — a late conversion, an upheld dispute — never an edit.

Config · 2

Build a rule — five slots

A guided slot-builder, not a blank editor. Each slot picks only from facts the estate actually captures — the surface itself is the guardrail. Anything with a pre-filled tag came from the Incentive Structure defaults and can be changed.
1

Condition — who and when the rule fires

Actor type, loan type, event — and the rule's type. e.g. SP + self-source + Fresh.
1 payee type selected
✦ Flash is a first-class rule type: additive over the base, this-period by default, auto-expiring. No engineering ticket per flash — same surface as any rule.
2

Basis — percent of what

The amount the rate applies to. Only facts we capture: ND from the loan record, A2NU% from the engine's own rollup.
⚠ Finance-metric basis (City Lead PAT bands, EBITDA kickers) is a documented fast-follow, not v1 — the slot shape holds, the feed ships just after.
3

Rate or slab — the number

A flat rate on the basis, or a slab table the period metric selects a row from.
For example — on ₹50,000 ND, this accrues ₹175
✦ The period metric selects the row — it does not gate and does not scale. Leave the last Up to blank for “and above”. (A2NU: select-slab → gate → scale.)
4

Modifiers — the gates and multipliers (optional)

Gate kind switches the rule on/off without changing the amount. Scale kind multiplies the computed payout. Compound gates compose on this one slot.
No modifiers — the rule pays its full rate to every accrual that matches the Condition.
5

Schedule — how it releases

The engine schedules tranches and marks each payable; a downstream payout action moves the money. The split and the 60-day clawback window are levers, not code.

Simulate the cost — the checker's evidence

The simulation agent replays the candidate over the last 3 months of real loans (backtest) and projects it forward against expected volume. Read-only: it never writes an accrual, never publishes. It reuses the live engine's exact aggregation paths, so it cannot drift from production.

Config · 6

Versioned publish

A rate change is a versioned publish with an effective-from date, never an in-place edit. Old accruals keep the version they were computed under, forever — as-of-event binding.
✓

Published — a new version is live

Effective from 01 Aug 2026. Accruals bind to this version as of their governing event — a loan disbursed under the old version keeps the old version, even if it's computed later.

version v8 status live effective 01 Aug 2026 rule RUL-2026-0714
Superseded, not deleted. The previous version in this category is marked superseded and stays on the record. For a period-grain rate, this change takes effect next period by default — the in-flight period is never disturbed.
